# Internal Auth Flow: Different Domain

Use this flow when the app is still a trusted internal app, but it runs on a different domain than Neup.Account.

## Important browser constraint

The main auth cookies are:

- `HttpOnly`
- `Secure`
- `SameSite=Lax`

Because of that, a browser page on another domain cannot read `auth_account_id`, `auth_session_id`, or `auth_session_key` from Neup.Account with JavaScript.

So the different-domain internal flow is not "read the cookies in the browser". The flow is "hand the validated session triplet to a trusted internal server, then verify it through the bridge".

## What this repo supports directly

Once your trusted server has the internal session triplet, this repo supports:

- `POST /bridge/api.v1/auth/verify`
- `POST /bridge/api.v1/auth/sign`
- gRPC `AuthService.VerifySession`

All three paths validate the same triplet:

```json
{
  "appId": "YOUR_INTERNAL_APP_ID",
  "appType": "internal",
  "auth_account_id": "acct_123",
  "auth_session_id": "sess_123",
  "auth_session_key": "key_123"
}
```

There is also a lighter validation mode with `appType: "fast"` when you only need a success/fail session check.

## Recommended flow

1. The user is already signed in on Neup.Account.
2. A trusted Neup.Account-controlled server context reads and validates the cookie session.
3. That trusted context passes the session triplet to your internal app server over HTTPS.
4. Your internal app server calls `POST /bridge/api.v1/auth/verify` or `POST /bridge/api.v1/auth/sign`.
5. Your internal app creates its own local session/cookie if needed.

The repo code shown here covers step 4 very clearly. Step 2 and step 3 are a deployment/infrastructure handoff problem, because browsers do not let another domain read these cookies directly.

## `verify` vs `sign`

### `POST /bridge/api.v1/auth/verify`

Use this when you only want to verify the triplet and get the basic user identity back.

Example response:

```json
{
  "success": true,
  "user": {
    "accountId": "acct_123",
    "displayName": "Jane Doe",
    "neupId": "jane.doe"
  }
}
```

### `POST /bridge/api.v1/auth/sign`

Use this when you want a bootstrap call for the internal app.

This route:

- validates the internal session
- creates an `app_authentications` row if one does not exist
- returns profile basics and app permissions

Example response:

```json
{
  "success": true,
  "accountId": "acct_123",
  "displayName": "Jane Doe",
  "displayImage": "https://...",
  "permissions": [],
  "isNewSignup": false
}
```

For `appType: "internal"` this route does not create an external `sessionValue`. That extra app session mapping is only created for `appType: "external"`.

## Server example

```ts
const response = await fetch('https://neupgroup.com/account/bridge/api.v1/auth/verify', {
  method: 'POST',
  headers: {
    'content-type': 'application/json',
  },
  body: JSON.stringify({
    appId: process.env.NEUP_APP_ID,
    appType: 'internal',
    auth_account_id: authAccountId,
    auth_session_id: authSessionId,
    auth_session_key: authSessionKey,
  }),
});

const data = await response.json();
```

## High-throughput option: gRPC

Internal apps can use the gRPC service in `src/grpc/services/auth-service.ts`.

Service definition:

```proto
service AuthService {
  rpc VerifySession (VerifyRequest) returns (VerifyResponse);
}
```

Use gRPC when:

- the caller is another trusted backend
- you want lower overhead than HTTP JSON
- you only need internal session verification

## A practical warning

This repo validates internal sessions very well, but it does not expose a public browser endpoint that safely "shares" `auth_*` cookies across different top-level domains.

That means your implementation should avoid:

- trying to read Neup.Account cookies in front-end JavaScript on another domain
- putting `auth_session_key` in URLs
- letting the browser become the trust bridge between domains

## Suggested architecture

- Neup.Account domain owns the real cookie session.
- Your internal app domain owns its own local app session.
- Server-to-server validation through `/bridge/api.v1/auth/verify`, `/bridge/api.v1/auth/sign`, or gRPC is the trust boundary between them.

## Signout guidance

For different-domain internal apps, sign out your local app session locally. If you also want to sign the user out of the master Neup.Account session, that has to happen on the Neup.Account side where the cookie-backed session actually lives.
