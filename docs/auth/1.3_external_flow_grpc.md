# External Flow gRPC

This document explains the current gRPC situation for external apps.

## Important clarification

The current repo does not implement a full external gRPC authentication flow.

Today, the external app flow is implemented with REST endpoints such as:

- handshake redirect
- grant exchange
- grant validation
- grant refresh
- profile fetch

Those flows are documented in:

- `docs/auth/1.3_external_flow.md`

## What gRPC currently supports

The current gRPC server exposes only one RPC:

```proto
service AuthService {
  rpc VerifySession (VerifyRequest) returns (VerifyResponse);
}
```

That RPC validates an internal session triplet:

- `appId`
- `auth_account_id`
- `auth_session_id`
- `auth_session_key`

This is an internal session verification RPC.

It is not a replacement for the external handshake + grant flow.

## What external apps should use today

For external apps, use REST:

1. redirect the browser to the handshake endpoint
2. receive `tempToken` and `authType`
3. exchange the `tempToken` with `POST /bridge/api.v1/auth/grant`
4. store `aid`, `sid`, and `skey`
5. validate and refresh through the existing REST grant routes

## Can another app call the current gRPC service?

Yes, but only if that app is acting like a trusted internal backend and already has the internal session triplet.

In that case it can call:

- `AuthService.VerifySession`

But that is no longer the normal external grant flow.

That is an internal trusted-backend use case.

## Current gRPC request and response

Request:

```proto
message VerifyRequest {
  string appId = 1;
  string auth_account_id = 2;
  string auth_session_id = 3;
  string auth_session_key = 4;
}
```

Response:

```proto
message VerifyResponse {
  bool success = 1;
  string error = 2;
  User user = 3;
}
```

Returned user fields:

- `accountId`
- `displayName`
- `neupId`

## Example Node.js gRPC client

This example is valid only for the internal verification RPC described above:

```ts
import * as grpc from '@grpc/grpc-js';
import * as protoLoader from '@grpc/proto-loader';
import path from 'path';

const PROTO_PATH = path.resolve(process.cwd(), 'src/grpc/proto/auth.proto');

const packageDefinition = protoLoader.loadSync(PROTO_PATH, {
  keepCase: true,
  longs: String,
  enums: String,
  defaults: true,
  oneofs: true,
});

const authProto = (grpc.loadPackageDefinition(packageDefinition) as any).auth;

const client = new authProto.AuthService(
  'neup-account-internal-host:50051',
  grpc.credentials.createInsecure()
);

client.VerifySession(
  {
    appId: 'trusted_internal_app',
    auth_account_id: 'acct_123',
    auth_session_id: 'sess_123',
    auth_session_key: 'key_123',
  },
  (error: Error | null, response: any) => {
    if (error) {
      console.error(error);
      return;
    }

    console.log(response);
  }
);
```

## If full external gRPC support is wanted later

A future external gRPC design would likely need separate RPCs for things like:

- exchange grant
- validate grant
- refresh grant
- fetch access snapshot

Those RPCs do not exist in the current repo yet.

## Summary

For external apps right now:

- use REST for handshake and grant
- use gRPC only if you are actually operating as a trusted internal backend with an internal session triplet

So the current gRPC implementation is internal-first, not external-first.
