# Internal Flow Different Domain gRPC

This document explains how a trusted internal app on another domain can use the Neup.Account gRPC service.

## Important browser limitation

A different-domain app cannot directly read Neup.Account auth cookies in the browser.

That means:

- the browser on another domain cannot read `auth_account_id`
- the browser on another domain cannot read `auth_session_id`
- the browser on another domain cannot read `auth_session_key`

So the gRPC caller must be a trusted backend that already received the session triplet through a safe server-side handoff.

## What the current gRPC service supports

The current service supports one RPC only:

```proto
service AuthService {
  rpc VerifySession (VerifyRequest) returns (VerifyResponse);
}
```

Request:

```proto
message VerifyRequest {
  string appId = 1;
  string auth_account_id = 2;
  string auth_session_id = 3;
  string auth_session_key = 4;
}
```

Response:

```proto
message VerifyResponse {
  bool success = 1;
  string error = 2;
  User user = 3;
}
```

Returned user fields:

- `accountId`
- `displayName`
- `neupId`

## Recommended different-domain flow

1. User is authenticated on Neup.Account.
2. A trusted Neup.Account-controlled server context validates the cookie-backed session.
3. That trusted context passes `auth_account_id`, `auth_session_id`, and `auth_session_key` to your internal app backend.
4. Your internal app backend calls Neup.Account gRPC `VerifySession`.
5. Your internal app backend continues with its own local session or business logic.

The key point is that the browser is not the trust bridge.

## How another internal app should call gRPC

The proto file used by the server is:

- `src/grpc/proto/auth.proto`

The service runs on:

- `GRPC_PORT`
- default `50051`

Current implementation detail:

- the server is started with `grpc.ServerCredentials.createInsecure()`

So this should be used on a private trusted network, service mesh, VPN, or behind a secured internal gateway.

## Example Node.js client

```ts
import * as grpc from '@grpc/grpc-js';
import * as protoLoader from '@grpc/proto-loader';
import path from 'path';

const PROTO_PATH = path.resolve(process.cwd(), 'src/grpc/proto/auth.proto');

const packageDefinition = protoLoader.loadSync(PROTO_PATH, {
  keepCase: true,
  longs: String,
  enums: String,
  defaults: true,
  oneofs: true,
});

const authProto = (grpc.loadPackageDefinition(packageDefinition) as any).auth;

const client = new authProto.AuthService(
  'neup-account-internal-host:50051',
  grpc.credentials.createInsecure()
);

client.VerifySession(
  {
    appId: 'your_internal_app',
    auth_account_id: authAccountId,
    auth_session_id: authSessionId,
    auth_session_key: authSessionKey,
  },
  (error: Error | null, response: any) => {
    if (error) {
      console.error(error);
      return;
    }

    console.log(response);
  }
);
```

## Example response

```json
{
  "success": true,
  "user": {
    "accountId": "acct_123",
    "displayName": "Jane Doe",
    "neupId": "jane.doe"
  }
}
```

## What to do after verification

After `VerifySession` succeeds, your app can:

- create or refresh its own local session
- look up the local account row by `accountId`
- store copied profile data if your integration wants that
- continue with internal permission or domain logic

## Security notes

- do not place `auth_session_key` in URLs
- do not let the browser on another domain read or relay the triplet directly
- call gRPC only from trusted backend code
- keep the gRPC server on private infrastructure unless you add proper transport security

## Summary

For different-domain internal apps, gRPC is supported only after a trusted server-side handoff:

- Neup.Account validates the cookie-backed session
- your backend receives the triplet securely
- your backend calls `VerifySession`
- your app gets `accountId`, `displayName`, and `neupId`
