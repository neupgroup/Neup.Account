# Internal Auth Flow: Same Domain

Use this flow when the application runs on the same site/cookie scope as Neup.Account and the browser can send the auth cookies automatically.

## Core idea

- The browser does not read the auth cookies directly.
- The browser sends them automatically on same-origin requests.
- The server validates the cookie triplet against the `sessions` table before trusting it.

In this repo the cookie-backed session helpers live in `src/lib/cookies.ts`, `src/lib/session.ts`, and `src/lib/auth-actions.ts`.

## Cookies used by the auth system

- `auth_account_id`: current signed-in personal account.
- `auth_session_id`: current session row id.
- `auth_session_key`: secret bound to that session.
- `auth_managing`: optional acting context such as `brand.<id>`, `dependent.<id>`, or `delegated.<id>`.
- `auth_accounts`: long-lived device account list for account switching. This is convenience state, not the proof of authentication.

These cookies are set with:

- `HttpOnly`
- `Secure`
- `SameSite=Lax`
- `path=/`

That means client-side JavaScript should not try to read them.

## How sign-in creates the session

`createAndSetSession()` in `src/lib/session.ts`:

1. creates a `sessions` row
2. generates `authSessionKey`
3. sets `auth_account_id`, `auth_session_id`, and `auth_session_key`
4. stores known accounts in `auth_accounts`

The main internal session lifetime is 30 days.

## Request flow

1. User signs in on Neup.Account.
2. The server sets the `auth_*` cookies.
3. The browser makes a same-origin navigation, form submit, server action, or API call.
4. The cookies are attached automatically by the browser.
5. Server code reads the cookies and validates them against the database.
6. Business logic runs only after validation succeeds.

`src/proxy.ts` only checks whether the cookies exist and redirects to `/auth/start` when they are missing. Real validation happens later with `getActiveSession()` or `validateCurrentSession()`.

## Recommended helpers

- `getActiveSession()`: validates `auth_account_id + auth_session_id + auth_session_key`.
- `validateCurrentSession()`: same as above, but redirects to `/auth/signout?...` when invalid.
- `getActiveAccountId()`: returns the managed account id when `auth_managing` is set, otherwise the personal account id.
- `getPersonalAccountId()`: always returns the personal signed-in account.
- `getSessionCookies()`: use only when you need raw cookie values.

## Server-side examples

### Server component

```ts
import { validateCurrentSession } from '@/lib/auth-actions';

export default async function Page() {
  const session = await validateCurrentSession();

  return <div>Signed in as {session.accountId}</div>;
}
```

### Route handler

```ts
import { NextResponse } from 'next/server';
import { getActiveSession } from '@/lib/auth-actions';

export async function GET() {
  const session = await getActiveSession();

  if (!session) {
    return NextResponse.json({ error: 'unauthorized' }, { status: 401 });
  }

  return NextResponse.json({
    success: true,
    aid: session.accountId,
  });
}
```

### Client request to your own API

```ts
await fetch('/api/internal/profile', {
  method: 'GET',
  credentials: 'include',
});
```

For same-origin requests the browser includes the cookies automatically, but keeping `credentials: 'include'` is fine when you want that behavior to stay explicit.

## When you need raw cookie values

Sometimes an internal endpoint or service needs the raw triplet:

```ts
import { getSessionCookies } from '@/lib/cookies';

const { accountId, sessionId, sessionKey, managingAccountId } =
  await getSessionCookies();
```

Use the raw values only on the server. Do not expose `sessionKey` to browser JavaScript unless there is a very specific trusted reason.

## Explicit validation endpoint

For internal session validation there is also:

- `POST /api.v1/auth/session`
- `DELETE /api.v1/auth/session`

`POST /api.v1/auth/session` accepts:

```json
{
  "aid": "USER_ACCOUNT_ID",
  "sid": "SESSION_ID",
  "skey": "SESSION_KEY",
  "deviceType": "web",
  "activity": "dashboard_open"
}
```

It validates the session and extends expiry by 30 days.

## Managing account context

If the user switches into a brand, dependent, or delegated account, the acting context is stored in `auth_managing`.

Use:

- `getPersonalAccountId()` when you need the real signed-in person
- `getActiveAccountId()` when you need the current acting account

## Refresh and signout

- `POST /bridge/api.v1/auth/refresh` extends the current cookie-backed session by 30 days.
- `logoutActiveSession()` expires the session row, removes linked `app_sessions`, updates `auth_accounts`, and clears the auth cookies.
- `/auth/signout` is the UI path that triggers that server action.

## Recommended rule of thumb

For same-domain internal apps:

- let the browser send cookies
- read them only on the server
- validate with `getActiveSession()` or `validateCurrentSession()`
- treat `auth_accounts` and `auth_managing` as supporting state, not primary authentication proof
