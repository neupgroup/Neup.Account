# External Auth Flow

Use this flow for third-party apps or any app that should not directly trust the main Neup.Account cookie session.

## Main flow

1. Redirect the browser to the handshake endpoint.
2. Neup.Account makes sure the user is signed in.
3. Neup.Account redirects back to your callback with a short-lived `tempToken`.
4. Your server exchanges that `tempToken` for an external app session.
5. Your server stores the returned session values securely and uses them for later validation and refresh.

## Step 1: Start the handshake

Browser redirect:

```txt
GET /bridge/handshake.v1/auth/grant?appId=YOUR_APP_ID&redirectsTo=https://yourapp.com/auth/callback
```

The handshake route:

- verifies the app exists
- checks whether the user already has a Neup.Account session
- redirects to `/auth/start` if they are not signed in
- creates a one-time `tempToken` valid for 5 minutes
- redirects back to your callback with:
  - `tempToken`
  - `authType` (`signin` or `signup`)

Example callback URL:

```txt
https://yourapp.com/auth/callback?tempToken=TOKEN_VALUE&authType=signin
```

## Step 2: Exchange the `tempToken`

This must be done from your server, not from browser JavaScript.

The repo exposes the same behavior at both:

- `POST /bridge/api.v1/auth/grant`
- `POST /api.v1/auth/grant`

Request:

```json
{
  "appId": "YOUR_APP_ID",
  "tempToken": "TOKEN_FROM_CALLBACK"
}
```

Response:

```json
{
  "aid": "acct_123",
  "sid": "ext_session_id",
  "skey": "ext_session_key",
  "jwt": "signed.jwt.value",
  "exp": 1710000000,
  "role": "user"
}
```

If the external role has extra permissions enabled, the response also includes `per`.

## Lifetime summary

- `tempToken`: 5 minutes, one-time use
- external `sid` + `skey`: 7 days
- JWT: 7 minutes

The JWT is signed with the application's `appSecret`.

## What gets stored on the Neup side

During the exchange flow:

- the one-time `tempToken` is stored in `session.dependentKeys`
- an `auth_sessions_external` row is created with:
  - `sid`
  - `skey`
  - `aid`
  - `appId`
  - `jwt`
  - `expiresOn`

## Step 3: Store the session on your side

Recommended:

- store `aid`, `sid`, and `skey` on your server or in your own HTTP-only cookie
- treat the JWT as a short-lived signed assertion, not your only durable session state
- never expose `appSecret` to the browser

## Step 4: Validate an existing external session

Use:

- `GET /bridge/api.v1/auth/grant`
- or `GET /api.v1/auth/grant`

Query params:

```txt
appId=YOUR_APP_ID
aid=USER_ACCOUNT_ID
sid=EXTERNAL_SESSION_ID
skey=EXTERNAL_SESSION_KEY
```

Example response:

```json
{
  "success": true,
  "aid": "acct_123",
  "appId": "your_app",
  "expiresOn": "2026-03-20T10:00:00.000Z",
  "lastLoggedIn": "2026-03-12T10:00:00.000Z"
}
```

## Step 5: Refresh the external session

Use:

- `PATCH /bridge/api.v1/auth/grant`
- or `PATCH /api.v1/auth/grant`

Request:

```json
{
  "appId": "YOUR_APP_ID",
  "aid": "USER_ACCOUNT_ID",
  "sid": "EXTERNAL_SESSION_ID",
  "skey": "EXTERNAL_SESSION_KEY"
}
```

Response:

```json
{
  "aid": "acct_123",
  "sid": "ext_session_id",
  "jwt": "new.jwt.value",
  "exp": 1710000420,
  "role": "user"
}
```

Refreshing extends the external session by another 7 days and issues a new 7-minute JWT.

## Optional profile fetch

`GET /api.v1/profile` supports two external auth modes:

### Persistent external session

Send headers:

```http
aid: USER_ACCOUNT_ID
sid: EXTERNAL_SESSION_ID
skey: EXTERNAL_SESSION_KEY
```

### One-time bootstrap using `tempToken`

Send query params:

```txt
tempToken=TOKEN_FROM_CALLBACK
appId=YOUR_APP_ID
```

When authenticated with `tempToken`, the endpoint can return the full self profile during the initial bootstrap.

## Security rules

- Do the handshake redirect in the browser.
- Do the `tempToken` exchange on the server.
- Keep `appSecret` server-only.
- Keep `sid` and `skey` in secure server-side storage or your own HTTP-only cookie.
- Expect the JWT to expire quickly and refresh it when needed.

## Legacy route note

This repo still contains an older external-app flow at:

- `POST /bridge/api.v1/auth/sign`
- `POST /bridge/api.v1/auth/signout`

That older flow creates an `app_sessions` row with a `sessionValue`.

For new integrations, prefer the handshake + grant flow documented above unless you are maintaining an older consumer that already depends on `sessionValue`.
