# Internal Flow Same Domain gRPC

This document explains how a same-domain internal app can use the Neup.Account gRPC service.

## What the current gRPC service does

The current gRPC service exposes one RPC:

```proto
service AuthService {
  rpc VerifySession (VerifyRequest) returns (VerifyResponse);
}
```

Request:

```proto
message VerifyRequest {
  string appId = 1;
  string auth_account_id = 2;
  string auth_session_id = 3;
  string auth_session_key = 4;
}
```

Response:

```proto
message VerifyResponse {
  bool success = 1;
  string error = 2;
  User user = 3;
}
```

The `User` payload contains:

- `accountId`
- `displayName`
- `neupId`

## When to use this

Use gRPC when:

- your app is a trusted internal backend
- your app is on the same domain or same trusted environment as Neup.Account
- your server can already read the cookie-backed session triplet
- you want backend-to-backend verification with lower overhead than REST

Do not call this directly from browser JavaScript.

The browser should talk to your app server first.

## Same-domain flow

1. User signs in on Neup.Account.
2. Browser sends the `auth_*` cookies to your same-domain server automatically.
3. Your server reads `auth_account_id`, `auth_session_id`, and `auth_session_key`.
4. Your server calls the Neup.Account gRPC `VerifySession` RPC.
5. The RPC returns the verified user identity.
6. Your app continues with its own business logic.

## How another app should call the gRPC service

The proto file in this repo is:

- `src/grpc/proto/auth.proto`

The server binds on:

- `GRPC_PORT`
- default port: `50051`

Current implementation detail:

- the server uses `grpc.ServerCredentials.createInsecure()`

So this should stay on a private trusted network unless you add transport security in front of it.

## Example Node.js client

```ts
import * as grpc from '@grpc/grpc-js';
import * as protoLoader from '@grpc/proto-loader';
import path from 'path';

const PROTO_PATH = path.resolve(process.cwd(), 'src/grpc/proto/auth.proto');

const packageDefinition = protoLoader.loadSync(PROTO_PATH, {
  keepCase: true,
  longs: String,
  enums: String,
  defaults: true,
  oneofs: true,
});

const authProto = (grpc.loadPackageDefinition(packageDefinition) as any).auth;

const client = new authProto.AuthService(
  'neup-account-internal-host:50051',
  grpc.credentials.createInsecure()
);

client.VerifySession(
  {
    appId: 'your_internal_app',
    auth_account_id: 'acct_123',
    auth_session_id: 'sess_123',
    auth_session_key: 'key_123',
  },
  (error: Error | null, response: any) => {
    if (error) {
      console.error(error);
      return;
    }

    console.log(response);
  }
);
```

## Example response

```json
{
  "success": true,
  "user": {
    "accountId": "acct_123",
    "displayName": "Jane Doe",
    "neupId": "jane.doe"
  }
}
```

## Where the session triplet comes from in same-domain apps

For same-domain internal apps, the server can read the cookie-backed session values using the existing server helpers in this repo.

Typical server-side helpers are:

- `getSessionCookies()`
- `getActiveSession()`

The important part is:

- read the triplet on the server
- send the triplet to gRPC from the server
- never expose `auth_session_key` to front-end JavaScript

## Security notes

- gRPC is currently internal/trusted-network oriented
- keep it behind private infrastructure or add TLS/proxy protection
- validate the app by sending the correct `appId`
- do not treat cookie existence alone as proof of authentication

## Summary

For same-domain internal apps, gRPC is a backend verification shortcut:

- browser sends cookies to your server
- your server reads the triplet
- your server calls `VerifySession`
- your app receives `accountId`, `displayName`, and `neupId`
